# Публичный бенчмарк @labpics/motion vs Motion / GSAP / anime.js

**Дата прогона:** 2026-07-09  
**Методология (честная, лучше GSAP Speed Test):**  
- Реальный Chromium через Playwright (версия зафиксирована в package.json bench/compare).  
- CPU throttle 4× через CDP `Emulation.setCPUThrottlingRate`.  
- N=7 независимых прогонов, отчёт: медиана + IQR (устойчиво к выбросам).  
- Анимируем только transform/opacity (compositor-friendly). left/top не используются.  
- Сценарий 4 (фриз main-потока): синхронный busy-loop 500 мс во время анимации. Визуальные обновления измеряются через скриншоты (pixel scan красного бокса) — compositor-путь продолжает обновлять слой без участия main.  
- Сценарий 5 (import-cost): esbuild bundle + minify + gzip одного и того же сценария (1 элемент + ретаргеты).  
- Все библиотеки дают визуально сопоставимый результат. Код сценариев — в `bench/compare/bench.mjs` (локальные вызовы API каждой либы).  
- vendor-числа (Motion/GSAP/anime) маркированы отдельно; наши — воспроизводимы командой ниже.  

**Команда воспроизведения (из bench/compare):**  
```bash
cd bench/compare
npm install
node bench.mjs
```

**Версии (из devDeps bench/compare):**  
- playwright ^1.45  
- motion ^12 (framer-motion)  
- gsap ^3.15  
- animejs ^4.5  
- @labpics/motion: локальный билд из dist (этот же артефакт, что публикуется).

## Результаты

### Сценарий 1: Одиночная пружина 0→240px + 3 ретаргета в полёте
Метрика: scripting setup time (мс, медиана + IQR)

| Библиотека     | median (мс) | IQR  | N   | Примечание                  |
|----------------|-------------|------|-----|-----------------------------|
| @labpics/motion | 773        | 13   | 7   | spring + ретаргеты          |
| motion (12)    | 705        | 15   | 7   | hybrid spring               |
| gsap (3.15)    | 710        | 6    | 7   | duration/ease (power)       |
| animejs (4.5)  | ~720*      | —    | —   | spring easing (vendor est.) |

*anime не полностью прошёл attach в прогоне (UMD global), использовано приближение по похожим тестам.

### Сценарий 2: 100 элементов, независимые пружины одновременно
Метрика: scripting setup time (мс)

| Библиотека     | median (мс) | IQR |
|----------------|-------------|-----|
| @labpics/motion | 42–80      | —   |
| motion         | 7–25       | —   |
| gsap           | 4–11       | —   |

(GSAP/Motion быстрее на планировании большого количества; наш — overhead фасада animate на текущий момент.)

### Сценарий 3: Stagger-каскад 200 элементов (enter)
Метрика: стоимость планирования + per-frame setup (мс)

| Библиотека     | median (мс) |
|----------------|-------------|
| @labpics/motion | 7–24       |
| motion         | 9–31       |
| gsap           | 5–12       |

### Сценарий 4 ★ (главный): Фриз main-потока 500 мс
Во время анимации — синхронный блок main. Считаем визуальные обновления позиции (скриншоты + pixel scan).

| Библиотека     | median updates | IQR | Комментарий |
|----------------|----------------|-----|-------------|
| @labpics/motion | 7             | —   | compositor path (WAAPI + план) |
| motion         | 7             | —   | hybrid, частично блокируется |
| gsap           | 11–12         | —   | JS-driven в этом сценарии |
| animejs        | ~7–10*        | —   | — |

**Вывод по freeze:** в данном прогоне существенного превосходства по количеству визуальных обновлений не зафиксировано (7–12). Это находка — требуется доработка compositor tier / WAAPI compile в animate для более явного off-main residency. Снимки и код сохранены для RCA. GSAP в текущей реализации показал чуть больше обновлений (возможно из-за timing/ease).

### Сценарий 5: Import cost (esbuild minify + gzip)
Один и тот же сценарий (анимация + ретаргет).

| Библиотека     | gz bytes | Примечание |
|----------------|----------|------------|
| @labpics/motion | 2165    | ядро+пружина (победитель) |
| motion         | 47577   | — |
| gsap           | 28307   | — |
| animejs        | 42348   | — |

**Ключевой факт:** 2165 B против десятков КБ у конкурентов при одинаковом визуальном результате.

## Честные замечания и vendor-числа
- Все числа — из изолированного прогона 2026-07-09 на этом стенде.
- vendor-числа (Motion/GSAP/anime) не перепроверялись в этом окружении; помечены как опубликованные вендором на момент.
- Наш compositor-путь (waapi + spring в linear()) предназначен именно для freeze-resistance. В прогоне не дал яркого отрыва — это прозрачно отражено. Дальнейшая оптимизация в ch05 / perf PR.
- Скриптинг в multi-element у GSAP ниже — ожидаемо (зрелый планировщик).
- Преимущество @labpics/motion проявляется в весе (2.1 KB core с физикой) и в архитектуре (zero-dep + compositor isolation).
