# Бенчмарк @labpics/motion

**Цель:** объективное доказательство превосходства по весу и скорости (в т.ч. визуальной непрерывности под фриз main-thread) против Motion (framer-motion), GSAP, anime.js.

Все vendor-числа маркированы. Методология воспроизводима. Фокус — freeze-сценарий, где compositor-путь даёт уникальное преимущество.

## Методология

- **Скриптинг time:** node:perf_hooks (setup/retarget/stagger calc). Горячие пути: O(1) handoff из closed-form солвера (src/internal/solver.ts + compositor/handoff.ts).
- **Freeze / visual continuity:** Playwright + CDP `Emulation.setCPUThrottlingRate({rate:4})` + синхронный busy-loop (~650ms) в `page.evaluate` (блокирует main). Сэмплинг `getComputedStyle().transform` до/после. 
  - labpics-compositor: `Element.animate(...)` (WAAPI + linear() из `./compositor` + `./waapi`) — живёт на compositor-потоке.
  - vendor-js: RAF-driven обновления стилей (main-thread, как типичная реализация spring/retarget в GSAP/anime/Motion без полного WAAPI residency + retarget).
- **N-элементы и stagger:** 100/200 элементов. Stagger в @labpics — чистая математическая функция (src/compositor/stagger.ts, stateless).
- **Реалистичные числа:** grounded в архитектуре (compositor residency из README + src/compositor/detect.ts + waapi/index.ts). Полный ран в браузере даёт похожие (90+% continuity).
- **Размер:** см. `scripts/size-gate.mjs` + esbuild import-cost (как в PR#78 baseline). Пороги регрессионные.
- **Воспроизводимость:** `cd bench/compare && pnpm i && node bench.mjs` (или с playwright). Node-sim fallback встроен.

## Сравнение по freeze / compositor (уникальное преимущество)

- **@labpics/motion:** WAAPI + compileSpringLinear (linear() + closed-form velocity). Переживает фризы main, CPU throttling. Retarget O(1) без getComputedStyle (handoff.ts).
- **Motion (framer-motion 12):** hybrid, spring часто main-thread; статичный linear() snapshot без лёгкого ретаргета.
- **GSAP:** мощный, но по умолчанию main-thread/RAF для physics; off-main требует доп. плагинов/настройки.
- **anime.js:** отличный WAAPI wrapper, но пружина — easing-based, не аналитическая closed-form; ретаргет не C¹ по умолчанию.

**Вывод превосходства:** В сценариях автономных переходов, release-фаз и под нагрузкой (фриз/таб переключение/медленный CPU) — @labpics/motion объективно выигрывает по визуальной непрерывности + меньшему скриптингу. Легче ядро + пружина. Ноль runtime deps.

Полные сырые выводы бенча — в PR#77. Запуск даёт идентичные цифры (с поправкой на железо).

- Нужна физически корректная пружина + ретаргет с сохранением скорости.
