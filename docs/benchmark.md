# Бенчмарки @labpics/motion

Этот документ описывает правила измерения. Числа не переписываются вручную:

- размер и import-cost вычисляет `scripts/size-gate.mjs`;
- сравнительный runtime-прогон генерирует отчёт в `bench/compare/results/`;
- исторический отчёт привязан к commit, версиям библиотек и окружению;
- регрессионные потолки живут рядом с исполняемым гейтом, а не копируются в документацию.

## Статус доказательств

| Область | Источник | Статус |
|---|---|---|
| Размер точек входа | `pnpm build && pnpm size` | Измеряется и защищён merge-гейтом |
| Import-cost потребителя | сценарии `scripts/size-gate.mjs` | Измеряется и защищён merge-гейтом |
| Массовый main-thread путь | `pnpm bench:ceiling` | p50/p95/p99 + структурный rAF-гейт |
| Runtime-сравнение | `bench/compare/bench.mjs` | Измеряется отдельным воспроизводимым прогоном |
| Сравнительный размер | `bench/compare/size-compare.mjs` | Измеряется воспроизводимым прогоном против пиненных версий |
| Числа конкурентов вне общего прогона | документация поставщика | Не считаются нашим измерением |

Performance-утверждение допустимо только вместе со сценарием, ревизией, версиями,
окружением, числом прогонов, способом агрегации и ограничениями методологии.

## Актуальное доказательство

Для `@labpics/motion@0.3.0` валидированного сравнительного отчёта нет; сравнительные результаты не заявлены.

## Сравнительный размер против Motion / GSAP / Anime.js

`bench/compare/size-compare.mjs` — воспроизводимый стенд import-cost
эквивалентного продуктового сценария (move+fade одного элемента) против
пиненных версий `motion` (mini и hybrid), `gsap` и `animejs` этого workspace:

```bash
pnpm build && cd bench/compare && pnpm install && node size-compare.mjs
```

Методология — та же, что у merge-гейта: esbuild bundle+minify против
установленных пакетов (lab-motion — из свежего `dist`), канонический
gzip-оракул; compiled-строка собирается реальным Vite с `motionCompiler()`.
Числа не копируются в Markdown: артефакт прогона с ревизией, версиями и
методологией — `bench/compare/size-compare.report.json`. Это сравнение цены
типового вызова, не утверждение о паритете возможностей (колонка contract
в отчёте фиксирует границы каждой строки).

## Размер и import-cost

### Что измеряется

`pnpm size` читает минифицированный `dist`, строит его статический граф импортов и
отдельно собирает потребительские сценарии. Гейт проверяет две разные величины:

1. **Shipped size** — сумма отдельно сжатых файлов начального статического графа
   точки входа, включая рекурсивные JavaScript-импорты и CSS `@import`.
2. **Consumer import-cost** — цена конкретного импорта после bundle, minify и
   tree-shaking в модели потребителя.

Вторая величина обязательна: маленький файл может случайно подтянуть большой
граф зависимостей, а общий размер `dist` не показывает цену отдельного сценария.
Все gzip-значения вычисляются одним детерминированным кодеком из
`scripts/compression-oracle.mjs`; размерный гейт и сравнительный S6 используют
его напрямую. Системный
Brotli остаётся вторым консервативным гейтом и привязан к зафиксированному Node executable.

### Законы гейта

- каждая публичная точка входа имеет собственный потолок;
- новый субпуть получает потолок от первой проверенной реализации с небольшим
  регрессионным запасом;
- потолок не повышается для прохождения CI без измеренного обоснования;
- сценарии точечных импортов доказывают точность `sideEffects`-allowlist и изоляцию модулей;
- сумма всех субпутей не используется как цена приложения: потребитель платит
  только за собственный граф импортов;
- снимок текущих чисел берётся из вывода команды, а не из таблицы в Markdown.

### Воспроизведение

```bash
pnpm install --frozen-lockfile
pnpm build
pnpm size
```

## Сравнительный runtime-прогон

Стенд `bench/compare` устанавливает реальные опубликованные пакеты, собирает
одинаковые сценарии и запускает их в Chromium через Playwright и CDP.

### Сценарии отчёта

- S1 — warm API-return одного элемента и cold CDP start marker→первый сдвинувшийся compositor-пиксель;
- S2 — cold/warm старт ста элементов одним вызовом;
- S3 — cold/warm stagger для двухсот элементов;
- S4 — cold/warm старт тысячи элементов одним вызовом;
- S5 — точность к собственной unblocked-траектории и видимое продвижение во время блокировки main thread;
- S6 — import-cost адаптера потребителя.

Cold realm в S1–S4 — новый JS realm с уже загруженным production-адаптером:
network/module fetch и parse не входят в эту метрику и не выдаются за startup
приложения. Null или ноль любой публикуемой cold-метрики останавливает отчёт;
стенд не заменяет потерянные наблюдения на «н/д».

Страница стенда приходит с COOP/COEP на loopback-origin и обязана подтвердить
`crossOriginIsolated`. Это снимает защитное огрубление `performance.now()` без
browser flags. Каждый pilot и publish-кластер сохраняет собственные raw-пробы
до и после измерения. Каждое утверждение использует максимум всех дельт своего
realm; справочное значение другого realm в порог не входит.

### Граница вывода freeze

Blocked-прогон сравнивается только с unblocked-прогоном той же библиотеки.
Linear-native и spring→WAAPI публикуются отдельными группами. Один невалидный
run или не движущийся платформенный контроль останавливает весь отчёт. Freeze-
прогоны допускаются только полными seeded-блоками: каждый участник проходит
каждую позицию ровно один раз за блок. Companion JSON хранит decoded baseline,
blocked и grid каждого run, достаточные для независимого пересчёта score.

### Проверка утверждений

Raw JSON хранит один кластер на независимый round-robin run; внутренние warm-
повторы не считаются отдельными наблюдениями. Отношение Lab/конкурент получает
воспроизводимый 95% paired cluster bootstrap CI. Множественные проверки образуют
одно семейство с поправкой Holm.

Победа по скорости публикуется, только если CI p50 целиком в пользу Lab,
выигрыш одновременно не меньше 5% и строго больше суммы худших realm-local
неопределённостей Lab и конкурента, все semantic-гейты зелёные, а верхняя
граница CI p95 не
выше 1,05. Пять процентов — продуктовая граница практической значимости, а не
статистический факт; тот же запас 5% для p95 — отдельная продуктовая политика
non-inferiority. Semantic-гейт вне timed-окна проверяет движение, конечность
координат и финал каждого target, включая всю топологию N=1000. Размер считается победой
только в одинаковой точной операции capability-группы и только при меньшем
каноническом gzip и системном Brotli.
Общего балла и рейтинга библиотек нет.

До сбора samples стенд общими pilot-раундами всех участников подбирает одно
число warm-вызовов на сценарий. Раунд удваивается, пока каждый повтор каждого
участника не достигнет четырёх локальных шагов сетки. Publish-batch проходит тот же
локальный гейт; равенство clock-порогу не доказывает различие.
Pilots не входят в выборку; raw-значения и каждый раунд сохраняются в companion JSON.
Шаг сетки для пола пакетного измерения берётся из верхнего края устойчивой
одношаговой моды. Дельты выше края обязаны лежать в диапазоне целого числа её
шагов; иначе отчёт отвергается. Причина пропуска шагов не выводится. Для порога
утверждений используется отдельно максимум всех исходных дельт до и после замера.

Для первого представленного пикселя CDP Runtime marker ставится непосредственно
перед API-вызовом. Он и screencast-кадр используют pinned CDP TimeSinceEpoch;
raw page-интервал marker→API входит в clock-порог.

Топология S1–S4 хранится одним калиброванным manifest в companion JSON. Из него
runner, semantic-validator и claim engine получают число вызовов, targets,
stagger и делитель абсолютного порога; отдельные копии запрещены.

### Ограничения

- API-return не равен полной стоимости анимации; первый представленный screencast-кадр измеряется отдельно;
- screencast доказывает browser-presented пиксель, но не input→photon физического дисплея;
- lazy-инициализация может переносить работу из вызова в первый кадр;
- физические модели пружин разных библиотек нельзя сравнивать как одинаковую
  траекторию без общего математического контракта;
- один компьютер и один браузер не задают универсальный рейтинг;
- wall-clock шум не используется как жёсткий merge-гейт;
- platform control обязан доказать distinct движение, иначе отчёт не создаётся;
- новые публичные claims требуют нового отчёта на актуальных ревизиях.

### Воспроизведение

```bash
cd bench/compare
pnpm install --frozen-lockfile
pnpm bench
```

Publish-прогон требует чистый commit, Node 24 и одинаковую точную версию pnpm в
корневом и benchmark `packageManager`. Стенд проверяет версии, хеширует фактически
установленные vendor-пакеты, Node executable, dist, адаптеры и browser binary, а
после прогона повторно проверяет runtime-артефакты.

S1–S4 используют не менее 20 позиционно-сбалансированных прогонов и
публикуют p50/p95. Freeze-матрица публикует только p50. p99 не публикуется,
пока каждая метрика не имеет не менее 100 независимых наблюдений.

Отдельный `node webkit-freeze.mjs` хранит видео, покадровые red/blue-позиции и
манифест в игнорируемом `reports/webkit-freeze/`. Манифест фиксирует фактические
версии и SHA-256 Playwright, esbuild, pngjs, browser executable, ffmpeg и адаптера;
после замера они хешируются повторно. Для долговременного аудита весь каталог
прикладывается к release issue или CI artifact, а не к Git-дереву.
Релизный workflow сам этот стенд не запускает, поэтому без приложенного каталога текущего
WebKit-доказательства нет.

## Потолок main-thread пути

`pnpm bench:ceiling` сам пересобирает текущий fingerprinted checkout и измеряет engine-only
путь без реального DOM/style/layout/paint. Он отвечает на четыре разных
вопроса:

- сколько занимает старт N=1/100/1000 целей для tween и spring;
- сколько занимает один paired-блок из 60 живых кадров тех же групп (p50/p95/p99);
- какую справочную долю бюджета 120/240 Гц занимает synthetic JS-хвост p99;
- сохраняется ли `1 requestFrame/кадр`, один уже queued drain после cancel и
  отсутствие повторных idle-заявок между разными entries пакета (фасад
  `./animate` + wc-биндинг на общем frame) и несколькими вызовами.

Каждый sample проверяет ровно одну запись каждого target в каждом кадре, scheduler,
`finished`/`onComplete`, queued drain и rolling hash полного property/value trace.
Независимый linear/underdamped oracle сверяет все 60 числовых значений; кадры
0/1/15/30/59 остаются компактными checkpoints в raw evidence. Только структурные
и семантические пункты являются жёстким гейтом: они не зависят от мощности CPU.
Абсолютная латентность служит профилем движка и не является браузерным frame
budget. Стенд также печатает горячие interior-запросы 1000-точечной интерполяции и
velocity-window при высокой частоте входных событий.

```bash
pnpm bench:ceiling
```

## Расширение стенда

Новый публичный контракт сопровождается отдельным benchmark-сценарием только
тогда, когда для него заявляется преимущество размера или производительности.
Сценарий сначала фиксируется в GitHub Issue с оракулом, контролем платформы и
ограничениями интерпретации; после реализации отчёт генерируется стендом.
