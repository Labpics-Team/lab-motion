# Методология размера: гейты, оракул сжатия, сравнение с конкурентами

> Роль: справка — объяснение того, как пакет измеряет и защищает вес. Числа
> здесь не приводятся намеренно: снимок всегда берётся из вывода команд.

## Закон: числа не живут в Markdown

Любое число размера устаревает в момент следующего коммита. Поэтому:

- фактические размеры печатает `pnpm build && pnpm size`;
- сравнение с конкурентами генерирует `bench/compare/size-compare.mjs`
  (артефакт с ревизией и версиями — `size-compare.report.json`);
- регрессионные потолки живут в исполняемом гейте `scripts/size-gate.mjs`
  рядом с хронологией изменений, а не в документации.

## Две разные величины

1. **Shipped size** — сумма отдельно сжатых файлов статического графа
   entry (как их отдаст CDN). Ловит физическое раздувание модуля.
2. **Consumer import-cost** — цена конкретного импорта после bundle +
   minify + tree-shaking в модели потребителя (esbuild против свежего
   `dist`). Ловит случайное втягивание чужого графа: маленький файл может
   стоить дорого, если тянет соседей.

Обе меряются на каждый прогон; сценарии import-cost включают точечные
импорты (страж tree-shakeability), полные фасады и совместные графы
(`animate + compositor`), потому что сумма изолированных сценариев ничего
не говорит о цене их сочетания.

## Канонический оракул сжатия

Разные zlib-реализации дают разные байты. Все gzip-значения считает один
детерминированный кодек (`scripts/compression-oracle.mjs`) — размерный гейт
и сравнительный стенд используют его напрямую, поэтому числа сопоставимы
между машинами и во времени. Системный Brotli — второй, консервативный гейт.

## Ратчеты «от факта»

Потолок каждого entry/сценария — регрессионный ратчет:

- новый субпуть получает потолок от факта первой реализации + небольшой люфт;
- потолок НЕ повышается ради прохождения CI — повышение требует явного
  решения владельца с хронологией в комментарии гейта;
- после оптимизаций потолок ЗАТЯГИВАЕТСЯ вниз к новому факту (это не
  ослабление — это фиксация выигрыша);
- рост допустим только как осознанная цена фичи и записывается в
  хронологию (дата, причина, факт, люфт).

Так вес может двигаться только вниз, а каждый байт роста имеет автора и
причину.

## Сравнение с конкурентами

`bench/compare/size-compare.mjs` собирает ОДИН продуктовый сценарий
(move+fade одного элемента) в идиоматической форме каждой библиотеки против
пиненных установленных версий, тем же esbuild-методом и тем же оракулом.
Это сравнение цены типового вызова, не паритета возможностей — колонка
contract в отчёте фиксирует границы каждой строки. Числа из чужих README
не используются никогда.

## Size-floor: несжимаемая часть контракта

`scripts/size-floor.mjs` (`pnpm size:floor`) вычисляет нижнюю границу веса
субпутя из несжимаемых токенов его поведенческого контракта: имена
платформенных API (минификатор не переименовывает свойства host-объектов и
глобалы), ключи публичной грамматики опций, контрактные CSS-строки и тексты
ошибок. Floor — canonicalGzip минимального модуля, содержащего каждый токен
по разу, тем же compression-оракулом, что merge-гейт; разница «факт − floor»
и есть цена собственной логики, единственная часть, которую имеет смысл жать.

Клейм строго «граница для данного контракта поведения», не «предел жанра»:
другой контракт (другие API, другая грамматика) имеет другой floor.
Честность манифеста охраняется машинно — каждый токен обязан буквально
присутствовать в шипнутом минифицированном субпуте, иначе скрипт падает.
Числа floor-а живут в выводе скрипта, не в этом документе.

## Воспроизведение

```bash
pnpm install --frozen-lockfile
pnpm build
pnpm size
pnpm size:floor
cd bench/compare && pnpm install --frozen-lockfile && node size-compare.mjs
```

Правила runtime-бенчей (CDP, cold/warm, freeze-матрица, статистика) — в
[docs/benchmark.md](../benchmark.md).
