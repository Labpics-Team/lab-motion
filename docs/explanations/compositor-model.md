# Фазовая модель main/compositor: куда уходит анимация

> Роль: справка — объяснение авто-tier и того, почему autonomous-движение не будит main-thread.

## Две фазы движения

**Фаза слежения** — значение ведёт пользователь: палец, курсор, скролл.
Здесь нужна обратная связь на каждый кадр — это законный main-thread
(`MotionValue`, `drive`, `./gestures`). **Автономная фаза** — release:
цель известна, траектория полностью описывается физикой. Считать её по
кадрам на main-thread не обязательно — браузер умеет исполнять готовую
кривую на compositor-потоке.

Пакет проводит эту границу автоматически: автономные переходы уходят на
compositor, интерактив остаётся живым.

## Как пружина становится compositor-кривой

Замкнутая форма пружины позволяет один раз (на компиляции, с кэшем)
превратить траекторию в кусочно-линейную кривую с ДОКАЗАННЫМ бюджетом
ошибки: адаптивный выбор узлов (Дуглас–Пекер по вертикальному отклонению)
ставит точки там, где кривизна высока, и разрежает экспоненциальный хвост.
Кривая сериализуется в CSS `easing: linear(...)`, план — два крайних кадра
`Element.animate()`. Дальше движение целиком исполняет браузер:
**ноль работы main-thread в steady-state, ноль rAF-подписок**. Блокировка
главного потока (тяжёлый JS, GC) не замораживает движение.

Подробнее о бюджете ошибки — [методология размера](./size-methodology.md) и
`src/compositor/segmenter.ts`; абсолютный бюджет в единицах свойства —
опция `maxValueError` (`compileSpringPlan`).

## Авто-tier: решение на вызов

Фасад `./animate` (и контроллер `CompositorSpring`) выбирает путь один раз,
в момент вызова, только по возможностям среды:

| Tier | Условие | Исполнение |
| --- | --- | --- |
| reduced | `prefers-reduced-motion` | мгновенный финал, ноль кадров |
| compositor | есть `Element.animate` + CSS `linear()` | WAAPI, 0 main-thread кадров |
| явные кадры | WAAPI без многостопового `linear()` (WebKit) | те же numeric-стопы явными кейфреймами |
| main (rAF) | нет WAAPI / инжектирован requestFrame / SSR-хост | живой цикл, та же замкнутая форма |

Детекция SSR-safe (без DOM на импорте) и мемоизирована. Какой tier выбран —
видно в диагностике контроллера (`CompositorSpring#tier`).

## Когда путь падает на rAF

- у цели нет `Element.animate` (duck-цель, тестовый хост);
- среда не исполняет `linear()` и не умеет явные кадры;
- вызывающий инжектировал собственный `requestFrame` (детерминированные
  тесты, кастомный цикл);
- импульс перехвата не представим сериализуемой кривой (искусственная
  solver-амплитуда вырожденного диапазона) — честный отказ, а не искажение
  траектории;
- бюджет кривой превышает физический кап сетки (`LM016`) — живой солвер
  дешевле и точнее.

Падение не меняет математику: обе ветки используют один солвер, финальные
значения и C¹-контракт совпадают (запечатано байт-паритетными тестами).

## Reduced-политика

`prefers-reduced-motion` — не «выключить анимации», а изменить характер:
финальное значение публикуется мгновенно, `finished`/`onComplete` работают,
реестр состояния обновляется. Функциональность (элемент оказался там, где
должен) сохраняется без движения. Шов `matchMedia` инжектируем.

## Nano и компилятор

`./nano` — тот же compositor-принцип в минимальном весе: spring → `linear()`
на лету, native Animation-контролы. Плагин `./compiler/vite` уносит и эту
работу из браузера: статические вызовы компилируются в готовый артефакт на
сборке (precomputed-кривая), рантайм-исполнителю остаётся только
`Element.animate` — ни солвера, ни парсера в бандле.
