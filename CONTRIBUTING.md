# Участие в разработке

## Назначение

Изменение принимается только вместе с доказательством пользовательского результата и сохранности доменных инвариантов.

Планы, приоритеты и статус работы живут в GitHub Issues. Документация описывает действующие контракты и способы работы.

## Рабочий цикл

1. Создать или уточнить issue:
   - пользовательский результат;
   - текущее наблюдаемое поведение;
   - ожидаемый контракт;
   - критерии приёмки;
   - не-цели;
   - риски совместимости.
2. Зафиксировать текущее поведение characterization-тестом, если оно неочевидно.
3. Добавить RED-тест, падающий по причине отсутствующего поведения.
4. Реализовать минимальное изменение до GREEN.
5. Удалить дублирование и укрепить границы модулей без изменения поведения.
6. Выполнить все обязательные гейты.
7. Обновить публичный контракт, документацию и changelog при пользовательском изменении.

## Классы тестов

| Класс | Назначение |
|---|---|
| Example | Конкретный пользовательский сценарий |
| Contract | Публичная поверхность, ошибки, lifecycle, post-destroy поведение |
| Property | Законы на диапазоне входов: финитность, монотонность, обратимость |
| Differential | Сравнение нового пути с независимым оракулом |
| Integration | Реальный артефакт, браузер, фреймворк или platform seam |
| Characterization | Защита существующего поведения перед рефакторингом |
| Mutation | Проверка силы оракулов критического доменного кода |

Тест не должен проходить при отсутствии проверяемой реализации. Исключения не проглатываются ради зелёного результата. Случайные входы имеют фиксированный seed и печатают его при сбое.

## Архитектурные границы

- Аналитическая физика и математика не зависят от DOM, фреймворков и глобального времени.
- Platform API доступны только через адаптер или инжектируемый seam.
- Субпуть владеет одним доменом и не экспортирует внутренние детали соседних модулей.
- Публичная поверхность запинена двусторонним contract-тестом.
- Один факт имеет один источник правды. Производные значения генерируются или проверяются drift-гейтом.
- Новая возможность расширяет существующий доменный примитив; параллельная реализация той же физики запрещена.
- Прерывание, отмена и уничтожение входят в основной контракт, а не считаются краевыми случаями.

## Производительность и размер

- Оптимизация начинается с воспроизводимого baseline.
- Изменение горячего пути сопровождается измерением до и после.
- Размерный бюджет выводится из фактической первой сборки и затем работает как регрессионный потолок.
- Сравнение с внешней библиотекой содержит версии, среду, сценарий, метод агрегации и ограничения.
- Машинозависимый wall-time не используется как единственный CI-гейт.

## Документация и комментарии

- Документация отвечает на один тип задачи: обучение, практическое руководство, справка или объяснение решения.
- Комментарий объясняет причину, ограничение или инвариант. Пересказ кода удаляется.
- Число, способное устареть, генерируется, проверяется гейтом или не дублируется.
- Временные заметки сессии, локальные ветки и незакоммиченный код не считаются состоянием проекта.

## Обязательные команды

```bash
pnpm install --frozen-lockfile
pnpm typecheck
pnpm build
pnpm test
pnpm size
pnpm pack:smoke
```

Для изменения критической математики:

```bash
pnpm mutation
```

## Pull request

PR содержит:

- ссылку на issue;
- пользовательский результат;
- границы изменения и не-цели;
- RED-доказательство или characterization baseline;
- результаты гейтов;
- измерения размера и производительности при затронутом горячем пути;
- сведения о breaking change, миграции и changelog;
- независимое adversarial review для нового доменного механизма.

Один PR закрывает один проверяемый контракт. Документация, рефакторинг и новая физика не смешиваются без общей причины изменения.
