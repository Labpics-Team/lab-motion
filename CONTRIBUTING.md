# Вклад в @labpics/motion

## Цель изменений

Каждое изменение должно улучшать наблюдаемое поведение потребителя и сохранять инварианты движка. Планы, очередность и статус работ ведутся в GitHub Issues. Документация описывает только действующие контракты.

## Архитектурные границы

- Доменная математика не зависит от DOM, фреймворков и глобального времени.
- Платформенные возможности подключаются через адаптеры и инжектируемые швы.
- Публичный API принадлежит домену субпути; внутренние детали не экспортируются.
- Один факт имеет один источник правды. Производные таблицы и манифесты генерируются или проверяются гейтом.
- Новая возможность не должна увеличивать неиспользуемые субпути: поставка остаётся модульной и `sideEffects: false`.
- Автономное движение использует compositor-путь, когда семантика платформы это допускает. Интерактивная follow-фаза остаётся на main-потоке.

## Рабочий процесс

1. Создайте или выберите issue с пользовательским результатом, инвариантами, критериями приёмки и не-целями.
2. Зафиксируйте текущее поведение characterization-тестом, если меняется существующий контракт.
3. Добавьте RED-тест, который падает по требуемой причине.
4. Реализуйте минимальное изменение без параллельного второго источника правды.
5. Выполните рефакторинг только при зелёном наборе тестов.
6. Обновите документацию действующего контракта и `CHANGELOG.md`, если изменение заметно потребителю.
7. Откройте небольшой PR с одним связанным пользовательским результатом.

История RED-фазы хранится в PR и коммитах. В коде и тестах остаются только устойчивые пояснения: какой закон защищён и почему он важен.

## Стратегия тестирования

Выбирайте минимальный уровень, способный обнаружить дефект:

- чистая математика — unit- и property-тесты с известными оракулами;
- публичная поверхность — contract/pin-тесты;
- взаимодействие модулей — sociable integration-тесты без моков собственного кода;
- DOM, WAAPI, layout и browser lifecycle — тесты в реальном браузере;
- исправление дефекта — регрессионный тест на класс поведения, а не только на одну строку;
- критические численные домены — seeded fuzz и mutation testing.

Не подменяйте браузер самодельным объектом там, где проверяется поведение браузера. Не используйте wall-clock как функциональный оракул: время должно быть виртуальным или инжектируемым.

## Производительность

Оптимизация принимается только вместе с:

- воспроизводимым сценарием;
- baseline и измерением после изменения;
- функциональным тестом, не позволяющим ускорить код ценой контракта;
- размерным бюджетом для затронутой точки входа;
- явными ограничениями методологии.

Сравнение с другими библиотеками выполняется на одинаковом сценарии, одинаковом окружении и реальных опубликованных пакетах. Симуляция не публикуется как измерение.

## Комментарии и документация

Комментарии объясняют причину, ограничение или инвариант. Пересказ кода удаляется.

Документация пишется на русском языке и разделяется по назначению:

- обучение — первый рабочий путь;
- практические руководства — решение конкретной задачи;
- справочник — точный контракт API;
- объяснение — архитектура и принятые компромиссы.

Динамические планы и статусы не дублируются в документации.

## Обязательные проверки

```bash
pnpm typecheck
pnpm build
pnpm test
pnpm size
pnpm pack:smoke
```

Для изменений численного ядра дополнительно:

```bash
pnpm mutation
```

## Требования к PR

В описании должны быть:

- пользовательский результат;
- изменённый контракт;
- доказательство RED → GREEN или characterization → изменение;
- риски и не-цели;
- результаты обязательных гейтов;
- данные замеров для performance-изменений;
- запись в changelog для заметного потребителю изменения.

Мерж запрещён при незакрытом P0/P1-дефекте в затронутом контракте, неразрешённом review thread или недостоверном performance-утверждении.
