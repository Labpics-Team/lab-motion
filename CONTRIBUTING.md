# Вклад в `@labpics/motion`

## Цель

Изменение принимается, когда улучшает наблюдаемое поведение потребителя и сохраняет инварианты движка. Один pull request решает одну причину изменения.

## Перед началом

Issue обязателен для:

- нового публичного API;
- изменения поведения или совместимости;
- межмодульной архитектуры;
- новых runtime-зависимостей;
- изменения размерного или производительного бюджета.

Небольшие исправления документации и локальные регрессии допускаются без отдельного issue, если PR содержит воспроизведение и критерий приёмки.

## Архитектурные границы

- Доменная математика не зависит от DOM, фреймворков и глобального времени.
- Платформенные операции находятся в адаптерах и вызываются через явные швы.
- Формула, токен, реестр экспортов и иной контракт имеют один источник правды.
- Новый субпуть отвечает одному домену и не дублирует соседний API.
- Скрытая связь через глобальное изменяемое состояние запрещена.
- Runtime-зависимость добавляется только при доказанном выигрыше, который нельзя получить внутри пакета дешевле.

## TDD

Рабочий цикл:

1. **RED** — тест падает по причине отсутствующего или ошибочного поведения.
2. **GREEN** — минимальная реализация восстанавливает контракт.
3. **REFACTOR** — структура улучшается без изменения наблюдаемого поведения.

RED-доказательство хранится в истории коммитов или описании PR. Комментарии в тестах объясняют закон и причину существования сценария, а не прошлое состояние реализации.

## Классы тестов

Выбирается минимальный набор, способный опровергнуть ошибочную реализацию:

- unit — локальная математика и ветвление;
- contract/pin — публичные экспорты, значения токенов, ошибки и упаковка;
- characterization — защита существующего поведения перед рефакторингом;
- property/fuzz — законы на пространстве входов, детерминированный seed;
- differential — сравнение с независимым оракулом;
- integration/browser — DOM, WAAPI, lifecycle, SSR и фреймворки;
- performance/size — только воспроизводимые бюджеты и сценарии потребителя;
- mutation — сила тестов критической доменной логики.

Тест, который не краснеет при целевой поломке, не считается доказательством.

## Инварианты движка

Каждое изменение проверяет применимые гарантии:

- наружу не выходят `NaN` и `Infinity`;
- одинаковый вход и clock дают одинаковый результат;
- импорт каждого субпути SSR-safe;
- прерывание сохраняет позицию и скорость, если контракт заявляет C¹;
- `reduced-motion` меняет характер движения предсказуемо;
- cancel, destroy и unmount не оставляют активных кадров, таймеров и подписок;
- неимпортированный модуль не увеличивает бандл потребителя;
- публичный API изменяется только явно и сопровождается миграцией.

## Обязательные проверки

```bash
pnpm install --frozen-lockfile
pnpm typecheck
pnpm build
pnpm test
pnpm size
pnpm pack:smoke
```

Для изменённого критического ядра дополнительно запускается целевой mutation-прогон. Для browser-кода требуется реальный браузерный тест, а не только DOM-эмуляция.

## Публичный API

При добавлении или изменении публичного контракта одновременно обновляются:

- исходный экспорт;
- `package.json#exports`;
- entry в `tsup.config.ts`;
- API surface pin;
- размерный бюджет от первого измеренного baseline;
- документация и пример;
- раздел `Unreleased` в `CHANGELOG.md`.

## Производительность

Оптимизация принимается только при наличии:

- воспроизводимого сценария;
- baseline до изменения;
- результата после изменения;
- проверки семантического паритета;
- регрессионного бюджета, если метрика стабильна между машинами.

Wall-clock измерение одной машины не используется как жёсткий CI-гейт без анализа шума.

## Документация и комментарии

- Документация — на русском языке.
- Комментарий объясняет, почему решение существует и какую ошибку предотвращает.
- Планы, прогресс и временные статусы находятся в Issues, а не в документации.
- Числа и списки, способные дрейфовать, генерируются или проверяются гейтом.

## Pull request

PR содержит:

- пользовательский результат;
- границы и не-цели;
- затронутые инварианты;
- RED-доказательство;
- риски и способ отката;
- фактические результаты проверок;
- changeset или запись в changelog для пользовательского изменения.

Самоодобрение не заменяет независимое ревью для публичного API, математики, безопасности, release-процесса и производительных заявлений.
